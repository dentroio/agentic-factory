# WO-1037 — Domain-Scoped Agent Runners

**Created:** 2026-07-19
**Priority:** P2
**Effort:** M
**Services:** agent-runner, orchestrator
**Depends on:** WO-1036

---

## Background

Running multiple agents in parallel on the same repo causes merge conflicts because every WO appends to the same "junction box" files (adapter.py, main.py, App.tsx). The structural fix (WO-407/408/409 in Clarion) eliminates the append pattern, but there is still a broader problem: agents are generalist and compete for any available WO regardless of which service it touches.

The architectural solution is domain-scoped agents: each runner instance declares which service domain it owns, and the orchestrator only dispatches WOs to matching runners. A frontend agent and a data-service agent never touch the same files, so they can safely run in parallel without conflicts.

## What to Build

### Agent runner: DOMAIN_FILTER env var

Add `DOMAIN_FILTER` to `config.py`:

```python
DOMAIN_FILTER = os.getenv("DOMAIN_FILTER", "")  # e.g. "frontend" or "data-service,src"
```

When `DOMAIN_FILTER` is set, the runner only claims WOs whose `services` field matches. Matching rule: any token in `DOMAIN_FILTER` appears as a substring of any token in the WO's `services` list (case-insensitive).

Examples:
- `DOMAIN_FILTER=frontend` claims WOs with `services: frontend` or `services: frontend,data-service`
- `DOMAIN_FILTER=connector-service` claims WOs with `services: connector-service`
- `DOMAIN_FILTER=` (empty, the default) claims any WO — existing behavior preserved

Add the filter check in `runner.py` before calling `/api/claim`. If the WO's services don't match, log `[runner] WO-NNN services={services} — not in domain {DOMAIN_FILTER}, skipping` and move to the next WO.

### Orchestrator: domain hints in dispatch

Add a `preferred_domain` field to the dispatch response so the status site can show which domain claimed a WO. No orchestrator-side routing logic needed — the agent-side filter is sufficient. Domain conflicts (two agents with overlapping domains) are handled by the claim race: whichever runner calls `/api/claim` first wins.

### LaunchAgent plists: four domain runners

Add four new plist templates to the factory's `scripts/` directory (generated by `agent-install.sh`):

| Label | DOMAIN_FILTER | PREFERRED_AGENT |
|---|---|---|
| `com.dentroio.factory-agent-frontend` | `frontend` | cursor |
| `com.dentroio.factory-agent-data` | `data-service` | cursor |
| `com.dentroio.factory-agent-connector` | `connector-service` | cursor |
| `com.dentroio.factory-agent-docs` | `docs,P3` | cursor |

The `docs` domain runner matches WOs whose services field is `none` or `docs`, or whose priority is `P3`. This serializes all docs-only WOs through a single runner, preventing concurrent doc edits.

The existing `com.dentroio.factory-agent-cursor` plist (no domain filter) becomes the generalist fallback — it runs when no domain runner is available.

### Status site

On the Factory tab status bar, show the active domain(s) instead of just "Cursor":
```
● 2 runners online  |  frontend · data  |  3 active
```

- [ ] `DOMAIN_FILTER=` runner claims any WO (existing behavior)

- [ ] WO with `services: frontend,data-service` is claimable by either domain runner
- [ ] Four domain plist templates exist in `scripts/`
- [ ] `agent-install.sh` installs domain plists alongside the generalist plist
- [ ] Status bar shows active domain labels
