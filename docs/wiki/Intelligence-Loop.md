# Intelligence Loop

The intelligence loop is a background job that runs every 10 minutes inside the orchestrator. It scans open PRs and dispatch state for problems and takes corrective action — closing bad PRs, creating fix WOs, re-triggering flaky CI, and clearing stale dispatch entries. The goal is to eliminate the class of interruptions where a human gets pulled in to handle something the factory should handle itself.

## What it does

### 1. Major-version Dependabot PRs → auto-close

Dependabot sometimes bumps a package across a major version boundary. These breaks are not safe to auto-merge — they almost always require code changes. The intelligence loop detects these by parsing the PR body for patterns like "Updates `foo` from 1.x to 2.x", then:

1. Posts `@dependabot ignore this major version` as a comment (suppresses future major bumps for that package)
2. Closes the PR

Minor-version Dependabot PRs are left alone; only cross-major-version bumps are closed.

### 2. Merge-conflict PRs → factory WO

When a PR's GitHub merge state is `dirty` (has a merge conflict), the intelligence loop:

1. Creates a factory WO (`WO-CONF-{N}`) in the plan queue at P1 priority
2. Posts a comment on the PR: `⚠️ Merge conflict detected — added to factory queue as WO-CONF-N (P1). An agent will rebase and resolve this shortly.`

The WO description is generated by Claude (Haiku) with instructions to fetch the branch, rebase onto main, resolve conflicts, and force-push. If no Anthropic key is available, a generic description is used.

### 3. Long-failing CI → LLM diagnosis

When a PR's CI has been failing for more than 90 minutes (configurable via `FAILURE_THRESHOLD_MINUTES`), the loop:

1. Fetches the failing check output from GitHub
2. Sends it to Claude (Sonnet) for diagnosis
3. Takes one of three actions based on the verdict:
   - **`create_wo`** — creates a factory WO with the diagnosed fix; posts a comment on the PR
   - **`rerun`** — re-triggers the failed check suites (for transient/flaky failures)
   - **`ignore`** — no action (CI is still running, or failure is expected)

### 4. Ghost dispatch entries → auto-clear

A "ghost" is a dispatch entry stuck in `claimed` or `in_progress` state where:
- The `last_seen` timestamp is older than 3 hours (configurable via `GHOST_THRESHOLD_HOURS`), AND
- No open branch exists on GitHub for that WO number

These are artifacts from agent processes that died mid-run without calling `/retry`. The loop marks them as `ghost` and clears them from the Active Agents panel.

## Deduplication

Each action is stamped with a 24-hour dedup key so the same PR or WO isn't acted on repeatedly within a day. The dedup state is in-memory and resets on container restart, which is intentional — restarting the orchestrator re-enables the loop to act on still-open issues.

## Configuration

| Environment variable | Default | Purpose |
|---------------------|---------|---------|
| `INTELLIGENCE_INTERVAL` | `600` | Seconds between passes |
| `FAILURE_THRESHOLD_MINUTES` | `90` | Minutes before CI failure triggers diagnosis |
| `GHOST_THRESHOLD_HOURS` | `3` | Hours before a stale claim is considered a ghost |

Set these in `.env.runtime` or your Docker environment.

## Monitoring

The factory floor's **Intelligence Loop** panel shows the last pass result: when it ran, how many PRs were scanned, what issues were found, and what actions were taken.

To check via API:
```bash
curl -s http://localhost:8100/api/intelligence/status | python3 -m json.tool
```

To trigger a manual pass immediately (without waiting for the next 10-minute tick):
```bash
curl -s -X POST http://localhost:8100/api/intelligence/run
```

The result is also persisted to `/data/intelligence_last_run.json` inside the orchestrator container, so it survives restarts and is visible immediately when the container comes back up.

## What it does NOT handle

- **PR review failures** — these are expected to require human or agent attention; the human checkpoint step in the WO flow handles them
- **Spec file status updates** — the orchestrator does not write back to WO spec files; those remain the agent's responsibility
- **Minor-version Dependabot bumps** — these are auto-merge eligible and handled by the normal Dependabot flow
- **Factory-paused state** — the intelligence loop still runs and creates WOs when the factory is paused; those WOs queue up and dispatch when the factory resumes
